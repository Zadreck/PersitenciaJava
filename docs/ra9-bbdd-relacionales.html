1. Introducción: de RA8 a RA9

En RA8 trabajamos con bases de datos orientadas a objetos (ObjectDB + JPA). En RA9 damos el salto a las bases de datos relacionales, que son las más extendidas en la industria web: MySQL, PostgreSQL, MariaDB, SQL Server, Oracle…

La buena noticia es que la mayoría de conceptos JPA que ya conoces se reutilizan directamente: @Entity, @Id, @OneToMany, JPQL… La diferencia está en la infraestructura (MySQL en lugar de un fichero .odb) y en el marco de trabajo (Spring Boot en lugar de un programa main() aislado).

¿Qué es Spring Boot?

Spring Boot es el framework Java más utilizado para desarrollar aplicaciones web y microservicios. Su principal ventaja es la auto-configuración: detecta automáticamente qué bibliotecas hay en el classpath y las configura con valores por defecto sensatos. Por ejemplo, si detecta spring-boot-starter-data-jpa y mysql-connector-j, configura automáticamente Hibernate como proveedor JPA y lo conecta al datasource que tú definiste en application.properties.

  • Starter POMs: dependencias empaquetadas temáticamente. spring-boot-starter-web incluye Tomcat embebido + Spring MVC. spring-boot-starter-data-jpa incluye Hibernate + Spring Data JPA.
  • Tomcat embebido: la aplicación lleva su propio servidor web. No hace falta desplegarla en un servidor externo; basta con ejecutar mvn spring-boot:run.
  • @SpringBootApplication: la anotación de arranque. Combina @Configuration, @EnableAutoConfiguration y @ComponentScan.
En este módulo usamos Spring Boot como backend de una API REST. Los datos se exponen como JSON a través de HTTP. En un proyecto real, el frontend (React, Angular, o una plantilla Thymeleaf) consumiría esos endpoints.

2. Configuración: Docker, MySQL y Spring Boot

2.1 MySQL en Docker

En lugar de instalar MySQL en el sistema operativo (lo que puede generar conflictos de versiones entre alumnos), lo ejecutamos en un contenedor Docker. Docker garantiza que todos trabajan con exactamente la misma versión y configuración de MySQL.

El fichero docker-compose.yml en la raíz del proyecto define el contenedor:

# docker-compose.yml
services:
  mysql_ra9:
    image: mysql:8.4
    container_name: mysql_ra9
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: rootpass
      MYSQL_DATABASE: ra9db
      MYSQL_USER: ra9user
      MYSQL_PASSWORD: ra9pass
    ports:
      - "3306:3306"
    volumes:
      - mysql_ra9_data:/var/lib/mysql      # datos persistentes entre reinicios
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  mysql_ra9_data:

Comandos esenciales de Docker:

2.3 application.properties

Sustituye al persistence.xml de RA8. Define la conexión a la base de datos y el comportamiento de Hibernate. La propiedad más importante para aprender es spring.jpa.hibernate.ddl-auto: 

# src/main/resources/application.properties

# ── Conexión a MySQL (Docker) ──────────────────────────────
spring.datasource.url=jdbc:mysql://localhost:3306/ra9db\
  ?serverTimezone=Europe/Madrid&useSSL=false
spring.datasource.username=ra9user
spring.datasource.password=ra9pass

# ── JPA / Hibernate ───────────────────────────────────────
spring.jpa.hibernate.ddl-auto=update   # crea tablas si no existen
spring.jpa.show-sql=true               # muestra el SQL generado por Hibernate
spring.jpa.properties.hibernate.format_sql=true
Valor ddl-autoComportamiento¿Cuándo usar?
noneNo toca el schemaProducción (schema gestionado aparte)
validateVerifica que el schema coincide con las entidadesProducción / staging
updateCrea tablas nuevas y añade columnas, no eliminaDesarrollo
createBorra y recrea todas las tablas al arrancarTests
create-dropCrea al arrancar y elimina al pararTests de integración
Con show-sql=true verás el SQL exacto que Hibernate envía a MySQL. Es la puente entre lo que conoces de JPA (RA8) y el SQL real de una base de datos relacional. Actívalo siempre durante el aprendizaje.

2.4 Primera entidad en MySQL

La clase principal de la aplicación RA9 es Ra9Application.java. Usa @EntityScan y @EnableJpaRepositories apuntando al paquete pio.daw.ra9 para que Spring Boot no intente gestionar las entidades de RA8 (ObjectDB): 

@SpringBootApplication
@EntityScan("pio.daw.ra9.modelo")
@EnableJpaRepositories("pio.daw.ra9.repositorio")
public class Ra9Application {
    public static void main(String[] args) {
        SpringApplication.run(Ra9Application.class, args);
    }
}

La entidad Pelicula usa las mismas anotaciones JPA que en RA8, pero ahora Hibernate crea una tabla real en MySQL: 

@Entity
@Table(name = "pelicula")
public class Pelicula {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // AUTO_INCREMENT en MySQL
    private Long id;

    @NotBlank(message = "El título no puede estar vacío")
    @Column(nullable = false)
    private String titulo;

    @Positive
    private Integer duracion;  // minutos

    @Column(name = "precio_entrada", precision = 6, scale = 2)
    private BigDecimal precioEntrada;

    // constructor vacío obligatorio para JPA (igual que en RA8)
    public Pelicula() {}
    // ... getters y setters
}

El repositorio es una interfaz que extiende JpaRepository. No hace falta implementar nada: Spring Data JPA genera automáticamente los métodos CRUD en tiempo de arranque. 

public interface PeliculaRepository extends JpaRepository<Pelicula, Long> {
    // Sin código: save(), findById(), findAll(), deleteById() ya están disponibles
}

El DataLoader implementa CommandLineRunner para insertar datos de prueba al arrancar: 

@Component
public class DataLoader implements CommandLineRunner {

    private final PeliculaRepository peliculaRepo;

    public DataLoader(PeliculaRepository peliculaRepo) {
        this.peliculaRepo = peliculaRepo;
    }

    @Override
    public void run(String... args) {
        if (peliculaRepo.count() > 0) return; // no recargar si ya hay datos
        peliculaRepo.save(new Pelicula("Inception", 2010, 148, new BigDecimal("9.50")));
        peliculaRepo.save(new Pelicula("Dune",      2021, 155, new BigDecimal("8.50")));
    }
}

3. API REST: CRUD con Spring Data JPA

3.1 Arquitectura por capas

Una aplicación Spring Boot bien estructurada separa el código en tres capas. Cada capa tiene una única responsabilidad y solo se comunica con la inmediatamente inferior:

CapaAnotaciónResponsabilidad
Controlador@RestControllerRecibe peticiones HTTP, valida entrada, devuelve respuesta JSON
Servicio@ServiceLógica de negocio, transacciones, llamadas a repositorios
Repositorio@Repository (implícito)Acceso a la base de datos mediante Spring Data JPA

3.2 @RestController y verbos HTTP

El @RestController mapea métodos Java a endpoints HTTP. La convención REST usa los verbos HTTP para expresar la operación: 

@RestController
@RequestMapping("/peliculas")
public class PeliculaController {

    private final PeliculaService peliculaService;

    public PeliculaController(PeliculaService peliculaService) {
        this.peliculaService = peliculaService;
    }

    // GET /peliculas  →  200 OK  →  lista de todas las películas
    @GetMapping
    public List<PeliculaDTO> listarTodas() {
        return peliculaService.listarTodas();
    }

    // GET /peliculas/1  →  200 OK  →  una película  |  404 si no existe
    @GetMapping("/{id}")
    public PeliculaDTO buscarPorId(@PathVariable Long id) {
        return peliculaService.buscarPorId(id);
    }

    // POST /peliculas  →  201 Created  →  película recién creada
    @PostMapping
    public ResponseEntity<PeliculaDTO> crear(@Valid @RequestBody Pelicula pelicula) {
        return ResponseEntity.status(HttpStatus.CREATED)
                             .body(peliculaService.guardar(pelicula));
    }

    // PUT /peliculas/1  →  200 OK  →  película actualizada
    @PutMapping("/{id}")
    public PeliculaDTO actualizar(@PathVariable Long id,
                                    @Valid @RequestBody Pelicula datos) {
        return peliculaService.actualizar(id, datos);
    }

    // DELETE /peliculas/1  →  204 No Content
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> eliminar(@PathVariable Long id) {
        peliculaService.eliminar(id);
        return ResponseEntity.noContent().build();
    }
}

El servicio actúa como intermediario entre controlador y repositorio:

@Service
@Transactional
public class PeliculaService {

    private final PeliculaRepository peliculaRepo;

    public PeliculaService(PeliculaRepository peliculaRepo) {
        this.peliculaRepo = peliculaRepo;
    }

    @Transactional(readOnly = true)
    public PeliculaDTO buscarPorId(Long id) {
        Pelicula p = peliculaRepo.findById(id)
            .orElseThrow(() -> new RecursoNoEncontradoException("Película no encontrada: " + id));
        return PeliculaDTO.desde(p);
    }
}

3.3 Manejo de errores HTTP con @RestControllerAdvice

En lugar de capturar excepciones en cada controlador, centralizamos el manejo en GlobalExceptionHandler. Spring invoca automáticamente el método correcto según el tipo de excepción: 

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(RecursoNoEncontradoException.class)
    public ResponseEntity<Map<String, Object>> handleNotFound(RecursoNoEncontradoException ex) {
        // devuelve JSON con status 404
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<...> handleValidacion(...) {
        // devuelve JSON con status 400 + lista de errores de validación
    }
}

4. Consultas con Spring Data JPA

Más allá de findAll() y findById(), Spring Data JPA ofrece tres mecanismos para consultas avanzadas que se corresponden con los que viste en RA8.

4.1 Métodos derivados por convención de nombres

Si el nombre del método sigue la convención de Spring Data, se genera el SQL automáticamente. No hace falta escribir ni JPQL ni SQL: 

public interface PeliculaRepository extends JpaRepository<Pelicula, Long> {

    // SELECT * FROM pelicula WHERE anio = ?
    List<Pelicula> findByAnio(Integer anio);

    // SELECT * FROM pelicula WHERE duracion < ?
    List<Pelicula> findByDuracionLessThan(Integer maxMinutos);

    // Navega la relación: JOIN director WHERE director.nacionalidad = ?
    List<Pelicula> findByDirectorNacionalidad(String nacionalidad);

    // LIKE case-insensitive
    List<Pelicula> findByTituloContainingIgnoreCase(String texto);
}
Las palabras clave de los métodos derivados incluyen: findBy, countBy, deleteBy, LessThan, GreaterThan, Between, Like, Containing, IgnoreCase, OrderBy, etc.

4.2 @Query con JPQL — misma sintaxis que RA8

Para consultas más complejas, usamos @Query. La sintaxis JPQL es exactamente la misma que aprendiste en RA8 con createQuery(): 

// En RA8 escribías:
TypedQuery<Pelicula> q = em.createQuery(
    "SELECT p FROM Pelicula p WHERE p.precioEntrada <= :max", Pelicula.class);
q.setParameter("max", max);

// En RA9 escribes en el repositorio:
@Query("SELECT p FROM Pelicula p WHERE p.precioEntrada <= :max ORDER BY p.precioEntrada ASC")
List<Pelicula> findBaratas(@Param("max") BigDecimal max);

// Agregados
@Query("SELECT AVG(p.precioEntrada) FROM Pelicula p")
BigDecimal avgPrecio();

@Query("SELECT COUNT(p) FROM Pelicula p")
Long totalPeliculas();

// SQL nativo cuando JPQL no basta
@Query(value = "SELECT * FROM pelicula WHERE anio BETWEEN :inicio AND :fin",
       nativeQuery = true)
List<Pelicula> findByRangoAnio(@Param("inicio") int inicio, @Param("fin") int fin);

4.3 Paginación con Pageable

Pageable sustituye a setFirstResult / setMaxResults de RA8. El controlador puede recibir los parámetros de paginación automáticamente desde la URL: 

// Repositorio
Page<Pelicula> findAll(Pageable pageable);

// Controlador
@GetMapping("/paginadas")
public Page<PeliculaDTO> paginadas(Pageable pageable) {
    return peliculaService.listarPaginadas(pageable);
}

// Llamada desde el cliente:
// GET /peliculas/paginadas?page=0&size=3&sort=titulo,asc

La respuesta incluye metadatos de paginación: número de página, total de elementos, total de páginas, etc.

5. Relaciones entre entidades

Las anotaciones de relaciones son exactamente las mismas que en RA8. La diferencia es que ahora Hibernate genera columnas de clave foránea reales en MySQL, que puedes ver con DESCRIBE pelicula; o SHOW CREATE TABLE pelicula;.

5.1 @OneToMany y @ManyToOne

Un Director tiene muchas Películas. En MySQL esto se traduce en una columna director_id en la tabla pelicula (clave foránea): 

// Director.java — lado "uno"
@Entity
public class Director {

    @OneToMany(mappedBy = "director", cascade = CascadeType.ALL,
               orphanRemoval = true, fetch = FetchType.LAZY)
    @JsonManagedReference  // evita bucle JSON al serializar
    private List<Pelicula> peliculas = new ArrayList<>();

    // método helper para mantener la coherencia bidireccional
    public void addPelicula(Pelicula p) {
        peliculas.add(p);
        p.setDirector(this);
    }
}

// Pelicula.java — lado "muchos"
@Entity
public class Pelicula {

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "director_id")  // columna FK en la tabla pelicula
    @JsonBackReference                  // no serializar esta dirección
    private Director director;
}
Sin @JsonManagedReference / @JsonBackReference (o DTOs), Jackson entraría en un bucle infinito al serializar: Director serializa sus Películas, cada Película serializa su Director, que serializa sus Películas… hasta StackOverflowError.

5.2 @ManyToMany con @JoinTable

Una película puede tener varios géneros, y un género puede pertenecer a varias películas. En MySQL esto requiere una tabla intermedia (pelicula_genero): 

// Pelicula.java
@ManyToMany(fetch = FetchType.LAZY)
@JoinTable(
    name = "pelicula_genero",
    joinColumns = @JoinColumn(name = "pelicula_id"),
    inverseJoinColumns = @JoinColumn(name = "genero_id")
)
private List<Genero> generos = new ArrayList<>();

// Genero.java (lado inverso)
@ManyToMany(mappedBy = "generos", fetch = FetchType.LAZY)
@JsonIgnore
private List<Pelicula> peliculas = new ArrayList<>();

Tras arrancar la app, verifica en MySQL: SHOW TABLES; debe mostrar la tabla pelicula_genero con dos columnas FK.

5.3 DTOs: la solución limpia al problema JSON

Un DTO (Data Transfer Object) es una clase sin anotaciones JPA que representa exactamente los datos que queremos enviar al cliente. Resuelve el problema del bucle JSON y permite controlar qué campos se exponen: 

public class PeliculaDTO {

    private Long id;
    private String titulo;
    private Integer anio;
    private String nombreDirector;   // solo el nombre, no el objeto Director
    private List<String> generos;    // solo los nombres de géneros

    public static PeliculaDTO desde(Pelicula p) {
        PeliculaDTO dto = new PeliculaDTO();
        dto.titulo = p.getTitulo();
        dto.nombreDirector = p.getDirector() != null
            ? p.getDirector().getNombre() + " " + p.getDirector().getApellidos()
            : null;
        dto.generos = p.getGeneros().stream().map(Genero::getNombre).toList();
        return dto;
    }
}

Los endpoints anidados permiten operar con recursos relacionados:

// GET /directores/1/peliculas  →  filmografía del director 1
@GetMapping("/{id}/peliculas")

                                                      @RequestBody Pelicula p) {

6. Transacciones y validación

6.1 @Transactional — gestión declarativa

En RA8 gestionabas las transacciones manualmente: 

// RA8 — gestión manual
EntityTransaction tx = em.getTransaction();
try {
    tx.begin();
    em.persist(objeto);
    tx.commit();
} catch (Exception e) {
    if (tx.isActive()) tx.rollback();
}

En RA9, Spring gestiona las transacciones automáticamente. Basta con anotar el método del servicio con @Transactional: Spring abre la transacción antes del método y hace commit al salir, o rollback si se lanza una excepción no controlada: 

// RA9 — gestión declarativa
@Transactional  // Spring abre tx → ejecuta método → commit o rollback automático
public EntradaDTO comprarEntrada(Long proyeccionId) {

    Proyeccion proyeccion = proyeccionRepo.findById(proyeccionId)
        .orElseThrow(() -> new RecursoNoEncontradoException(...));

    if (proyeccion.getAsientosDisponibles() <= 0) {
        // lanzar excepción provoca rollback automático
        throw new SinAsientosException(proyeccionId);
    }

    // decrementa asientos y persiste la entrada en la misma transacción
    proyeccion.setAsientosDisponibles(proyeccion.getAsientosDisponibles() - 1);
    proyeccionRepo.save(proyeccion);

    Entrada entrada = new Entrada(proyeccion.getPelicula().getPrecioEntrada(), proyeccion);
    entradaRepo.save(entrada);

    return EntradaDTO.desde(entrada);
}

Si SinAsientosException es lanzada, Spring hace rollback: el decremento de asientos no se persiste en MySQL. Las propiedades ACID quedan garantizadas por MySQL y por @Transactional.

6.2 Bean Validation en entidades y controladores

Las anotaciones de validación que pusiste en las entidades (@NotBlank, @Positive, @DecimalMin…) se activan cuando el controlador anota el parámetro con @Valid: 

// Controlador: @Valid dispara la validación antes de entrar al método
@PostMapping
public ResponseEntity<PeliculaDTO> crear(@Valid @RequestBody Pelicula pelicula) {
    return ResponseEntity.status(201).body(peliculaService.guardar(pelicula));
}
